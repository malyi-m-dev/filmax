# Filmax — правила рефакторинга и читаемости

> Черновик. Цель — единый свод, по которому прогоняем код: **меньше вложенности, короче методы,
> читаемее без потери смысла**. Правила — ориентиры, а не догма: если следование правилу делает
> код хуже, побеждает читаемость (и это поясняем комментарием).

## 0. Границы (что рефакторинг НЕ делает)

- **Не меняет поведение.** Только форма кода. Бизнес-логика, тексты, числа, порядок вызовов — как есть.
- **Не трогает публичные API** (сигнатуры `public`/модульных функций, DTO-поля, маршруты) без отдельного согласования.
- **Сохраняет KDoc-комментарии «почему».** В проекте комментарии объясняют неочевидные решения — это ценность, их переносим/обновляем, а не выкидываем.
- **Каждый модуль после прохода компилируется**, существующие проверки зелёные. Один модуль — один логический коммит.

---

## 1. Плоскость: минимум вложенности

Целевая глубина вложенности в теле функции — **≤ 3** уровней.

- **Guard clauses вместо пирамид.** Невалидное/краевое — отсекаем в начале ранним `return`/`?: return`.
  ```kotlin
  // было
  fun handle(item: Item?) {
      if (item != null) {
          if (item.isReady) {
              process(item)
          }
      }
  }
  // стало
  fun handle(item: Item?) {
      val ready = item?.takeIf { it.isReady } ?: return
      process(ready)
  }
  ```
- **Нет `else` после `return`/`throw`** — убираем лишний уровень отступа.
- **`when` вместо лестницы `if/else if`** на 3+ ветках.
- **Вложенный блок (цикл/тяжёлая лямбда) → отдельная приватная функция** с говорящим именем.
- **Цепочка `?.let { ?.let { … } }` — запах.** Одно значение → `?: return`/`?:`. Несколько → `when`/ранние выходы.
- **Коллекции — операторами, а не ручными циклами** с накоплением: `map`/`filter`/`firstOrNull`/`associate`/`groupBy` вместо `for` + `if` + `add`.

## 2. Методы: короче и об одном

- **Одна функция — одна задача.** Имя описывает задачу целиком; если в имени напрашивается «и» — делим.
- **Ориентир длины тела ≤ ~30 строк** (Composable — ≤ ~50, но дробим на под-`@Composable`). Это ориентир для внимания, не жёсткий предел.
- **Single-expression там, где читаемо:** `fun word(n: Int): String = when { … }`.
- **Сложное выражение → именованный `val`.** Условие/арифметика с «магией» получает имя, раскрывающее смысл.
- **Параметров > 5 → объект-параметр** (`data class`/`State`). Особенно для Compose-секций — передаём срез state, а не десяток примитивов.
- **Глубоко вложенные лямбды в Composable → выносим** в именованную функцию-секцию.

## 3. Читаемость и именование

- **Без однобуквенных имён** (`s`, `f`, `c`). Полное имя или `it` в коротком scope. (см. память проекта)
- **Булевы — `is/has/should/can`**, формулировка положительная: `isVisible`, не `isNotHidden`.
- **Магические литералы → именованные константы.** В Compose: повторяющиеся `dp/sp/Color` — в константы файла или в тему/дизайн-систему, а не россыпью по дереву.
- **Долой `!!`.** Заменяем на `?:` (дефолт/ранний выход) или `requireNotNull(x) { "понятное сообщение" }`.
- **Имя про «что», а не «как».** `visibleItems`, не `filteredList2`.

## 4. Идиоматичный Kotlin

- `?:` для дефолтов и ранних выходов; `when` как выражение; деструктуризация.
- `apply/also/run/with` — для конфигурации/побочных эффектов, **не** ради сжатия в одну строку.
- Повторяющееся преобразование → **extension-функция** рядом с типом.
- `data class`/`value class` для группировки связанных значений вместо «параллельных» переменных.

## 5. Compose-специфика

- **Большой `@Composable` → дерево мелких именованных.** Каждая визуальная секция — своя функция.
- **Хардкод `dp/sp/Color` → константы/тема.**
- **State hoisting:** дочерние — по возможности stateless, состояние поднимаем.
- **`remember` для дорогих вычислений** в теле composable (списки, форматирование, парсинг).
- **Передаём срез state**, а не весь объект, чтобы сузить рекомпозицию.

## 6. Чек-лист прохода по модулю

1. Функции > ~40 строк — разбить/выпрямить.
2. Вложенность > 3 — guard clauses, вынос блоков.
3. `!!`, магические литералы, однобуквенные имена — убрать.
4. Ручные циклы накопления — на операторы коллекций.
5. Толстые Composable — декомпозировать; хардкод-размеры — в константы.
6. KDoc-комментарии «почему» — на месте и актуальны.
7. `./gradlew :<module>:compile...Kotlin` зелёный; поведение не изменилось.

## 7. Процесс прогона

- **Инкрементально, по модулю.** Порядок: переиспользуемое ядро → фичи (эффект ядра множится).
- **Пилот на одном модуле**, ревью диффа, калибровка правил — и только потом масштабирование.
- Отдельный коммит на модуль, говорящее сообщение (`refactor(core:ui): …`).
- При сомнении «форма vs поведение» — выбираем сохранение поведения и помечаем место для обсуждения.
